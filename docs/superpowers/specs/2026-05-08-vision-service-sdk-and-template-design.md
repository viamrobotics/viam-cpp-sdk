# Vision service: SDK class + module-generator template

**Date:** 2026-05-08
**Status:** Approved (pending spec review + user review)
**Repo:** viam-cpp-sdk

## Background

The C++ SDK is missing a Vision service. The proto stubs at `src/viam/api/service/vision/v1/` exist (auto-generated `viam::service::vision::v1::VisionService`), but there is no `viam::sdk::Vision` abstract base class, no client/server pair under `src/viam/sdk/services/private/`, and no module-generator template under `res/module_generator/_templates/services/`. Peer services (`mlmodel`, `motion`, `navigation`, `discovery`, `generic`) all have the full triplet: SDK class, client/server, generator template.

The user wants a generator template for vision modules. The template depends on a `viam::sdk::Vision` class to inherit from, so the template alone is insufficient — both layers ship together.

## Goals

- Add `viam::sdk::Vision` as a full peer to existing service classes (mlmodel/motion/etc.). Cover all 9 proto RPCs in one PR.
- Provide a module-generator template that scaffolds a working Vision module skeleton.
- Maintain consistency with existing SDK patterns: hide proto types behind POD structs, use `ProtoStruct` for `extra`, mirror the file layout / CMake wiring of `mlmodel`.
- Keep the public API additive — no changes to existing types or generated proto code.

## Non-goals

- Fixing the pre-existing template-generation bug for `mlmodel` / `generic` (their templates emit `viam::sdk::Mlmodel` / `viam::sdk::Generic` which don't compile because the real classes are `MLModelService` / `GenericService`).
- An example vision module under `src/viam/examples/`. The generator template is the canonical example.
- Async / streaming RPC variants. Proto is unary; not needed.
- Cross-SDK (Go/Python) naming audit beyond what the proto already enforces.
- A version bump in this PR. Version bumps are separate commits per repo convention.

The workflow runs `cl_gen` against `src/viam/sdk/services/vision.cpp` whenever the header changes on `main`. Because the new class is named `Vision` (matching the file stem `vision`), the generator's stem→PascalCase derivation produces a working `viam::sdk::Vision` reference. Hand-writing the templates is a fallback only if a local `cl_gen` smoke-test fails.

### Class naming

Use `viam::sdk::Vision` (bare name, matching `Motion` / `Navigation` / `Discovery`). This avoids the `MLModelService` / `GenericService` template-generation issue without us having to fix that issue here.

## Public API surface

`Vision` lives in `src/viam/sdk/services/vision.hpp`, inherits from `viam::sdk::Service`, and exposes nine virtual methods matching the proto. Each takes an `extra` `ProtoStruct` consistent with peer services (e.g. `MLModelService::infer`, `Motion::move`), plus a no-extra convenience overload.

### Payload types (nested in `Vision`)

```cpp
struct detection {
    boost::optional<std::int64_t> x_min, y_min, x_max, y_max;       // pixel bbox (proto: optional int64)
    boost::optional<double> x_min_normalized, y_min_normalized,
                            x_max_normalized, y_max_normalized;      // normalized bbox (proto: optional double)
    std::string class_name;                                          // proto: plain string
    double confidence;                                               // proto: plain double
};

struct classification {
    std::string class_name;
    double confidence;
};

using point_cloud = Camera::point_cloud;   // mime_type + bytes; reuses Camera component type
using raw_image  = Camera::raw_image;      // for capture_all_from_camera

struct point_cloud_object {
    point_cloud point_cloud;                       // raw bytes (PCD or similar)
    std::vector<GeometryConfig> geometries;        // existing viam::sdk::GeometryConfig
};
// Note: the field name shadows the type name. This is legal C++ and matches how
// the proto field is named. Flag this explicitly in the PR description so reviewers
// don't take it for an oversight.

struct properties {
    bool classifications_supported;
    bool detections_supported;
    bool object_point_clouds_supported;
};

struct capture_options {
    bool return_image = false;
    bool return_detections = false;
    bool return_classifications = false;
    bool return_object_point_clouds = false;
};

struct capture_all_result {
    boost::optional<raw_image> image;
    std::vector<detection> detections;
    std::vector<classification> classifications;
    std::vector<point_cloud_object> objects;
    ProtoStruct extra;
};
```

### Virtual methods

```cpp
virtual std::vector<detection>           get_detections_from_camera(const std::string& camera_name, const ProtoStruct& extra) = 0;
virtual std::vector<detection>           get_detections(const raw_image& image, const ProtoStruct& extra) = 0;
virtual std::vector<classification>      get_classifications_from_camera(const std::string& camera_name, int count, const ProtoStruct& extra) = 0;
virtual std::vector<classification>      get_classifications(const raw_image& image, int count, const ProtoStruct& extra) = 0;
virtual std::vector<point_cloud_object>  get_object_point_clouds(const std::string& camera_name, const std::string& mime_type, const ProtoStruct& extra) = 0;
virtual struct properties                get_properties(const ProtoStruct& extra) = 0;
virtual capture_all_result               capture_all_from_camera(const std::string& camera_name, const capture_options& opts, const ProtoStruct& extra) = 0;
virtual ProtoStruct                      do_command(const ProtoStruct& command) = 0;
virtual ProtoStruct                      get_status() = 0;
```

Each gets an inline no-extra overload (`{ return foo(args, {}); }`), matching the `MLModelService::infer` pattern.

The public API is proto-free. `point_cloud_object::point_cloud` reuses `Camera::point_cloud` which already wraps bytes.

## Proto ↔ POD conversions

A single TU `src/viam/sdk/services/private/vision.cpp` (paired with `vision.hpp` in the same directory) holds free functions in `namespace viam::sdk::impl::vision`, used by both client and server. This mirrors `services/private/mlmodel.cpp` (260 LoC, which uses the parallel `viam::sdk::impl::mlmodel` nested namespace).

```cpp
viam::service::vision::v1::Detection             to_proto(const Vision::detection&);
Vision::detection                                from_proto(const viam::service::vision::v1::Detection&);

viam::service::vision::v1::Classification        to_proto(const Vision::classification&);
Vision::classification                           from_proto(const viam::service::vision::v1::Classification&);

viam::common::v1::PointCloudObject               to_proto(const Vision::point_cloud_object&);
Vision::point_cloud_object                       from_proto(const viam::common::v1::PointCloudObject&);

viam::service::vision::v1::GetPropertiesResponse to_proto(const Vision::properties&);
Vision::properties                               from_proto(const viam::service::vision::v1::GetPropertiesResponse&);

viam::service::vision::v1::CaptureAllFromCameraResponse to_proto(const Vision::capture_all_result&);
Vision::capture_all_result                              from_proto(const viam::service::vision::v1::CaptureAllFromCameraResponse&);
```

### Conversion details

- **`raw_image`:** Camera's existing client/server inline the field-by-field conversion rather than calling shared `to_proto`/`from_proto` helpers — there is no helper to "reuse" today. We will introduce one as part of this work by lifting Camera's inline conversion into a new shared private header `src/viam/sdk/common/private/raw_image.hpp` (paired with `raw_image.cpp`), consumed by both camera and vision. Lives under `common/private/` rather than `components/private/` because it's not Camera-specific anymore once Vision uses it. The lift is mechanical (~30 LoC).
- **Optional bbox fields on `detection`:** the proto presence-tracks pixel coords (`optional int64 x_min ...`) and normalized coords (`optional double x_min_normalized ...`) separately (8 optional fields in two groups of 4). `from_proto` uses `has_*` checks per field; `to_proto` only sets fields whose `boost::optional` is engaged. No silent zero-fill. `class_name` and `confidence` are plain (non-optional) fields on the proto, so they map to plain `std::string` / `double` on the POD.
- **`extra`:** every request/response that carries a `google.protobuf.Struct extra` round-trips through the existing `ProtoStruct ↔ google::protobuf::Struct` helpers in `viam::sdk::common`, the same way `mlmodel_client.cpp` handles `infer`'s `extra`.

## Client / Server plumbing

Two TUs, each ~150–200 LoC, mirroring `mlmodel_client.cpp` / `mlmodel_server.cpp`.

### `private/vision_client.{hpp,cpp}`

```cpp
class VisionClient final : public Vision {
public:
    VisionClient(std::string name, std::shared_ptr<grpc::Channel> channel);

    // Implements all 9 virtual methods. Each ~10–15 lines:
    //   1. build the request proto via to_proto helpers + extra→Struct
    //   2. invoke stub_->RPC(&ctx, req, &resp)
    //   3. on non-OK grpc::Status, throw viam::sdk::GRPCException
    //   4. convert response via from_proto helpers

private:
    std::unique_ptr<viam::service::vision::v1::VisionService::StubInterface> stub_;
};
```

Registration is added to `Registry::register_resources()` in `src/viam/sdk/registry/registry.cpp` as a single line `register_resource<impl::VisionClient, impl::VisionServer>();`, alphabetically placed near the existing `MLModelServiceClient` / `MotionClient` registrations (around line 229).

### `private/vision_server.{hpp,cpp}`

```cpp
class VisionServer final : public ResourceServer,
                            public viam::service::vision::v1::VisionService::Service {
public:
    explicit VisionServer(std::shared_ptr<ResourceManager> manager);

    grpc::Status GetDetectionsFromCamera(...);
    // ... one override per RPC
};
```

Each override:
1. Looks up the named resource via `ResourceManager`.
2. `dynamic_cast` to `Vision&`. On failure → `grpc::Status(INVALID_ARGUMENT, ...)`, matching the existing pattern.
3. Converts `extra` proto → `ProtoStruct`.
4. Calls the corresponding virtual on the user impl.
5. Catches `viam::sdk::Exception` and unknown exceptions, mapping to `grpc::Status` (mirror `mlmodel_server.cpp`'s catch ladder verbatim).
6. Converts the result back to proto and returns `grpc::Status::OK`.

`do_command` and `get_status` use the existing shared helpers (`viam::sdk::common::do_command` and the `Service`-provided `GetStatus` machinery) rather than rolling their own. Note that the exact split between shared helper vs per-service implementation varies across peer services; mirror whichever pattern `mlmodel_server.cpp::GetStatus` uses verbatim — verify at planning time.

### Threading / lifecycle

No new requirements. The existing `Service` base handles lifecycle. Each override is synchronous and runs on the gRPC thread pool, identical to other services.

## Test strategy

Mirror `test_mlmodel.cpp` and the existing `mocks/` pattern.

### `mocks/mock_vision.{hpp,cpp}`

Concrete `MockVision : public Vision` with:
- Constructor accepts canned response data (`std::vector<detection>`, `properties`, etc.) so each test can stand up a mock for the RPC under test.
- Each virtual records request fields (`last_camera_name`, `last_count`, `last_extra`, ...) and returns the corresponding canned data. `do_command` echoes its argument.
- Two-line `static std::shared_ptr<MockVision> make(...)` factory consistent with `mock_mlmodel`.

### `test_vision.cpp`

One Boost.Test suite per RPC. For each RPC:

1. Construct `MockVision` with canned data.
2. Stand up a `VisionServer` over an in-process gRPC channel; register the mock as the named resource.
3. Construct a `VisionClient` against the same channel.
4. Call the client method. Assert:
   - canned response round-trips intact (no field loss across proto conversion),
   - mock observed expected request fields,
   - `extra` survives the round-trip.

### Additional cases beyond happy-path round-trips

- `detection` with both pixel and normalized bbox fields engaged → both round-trip.
- `detection` with neither group engaged → neither set on proto, neither set on POD coming back.
- `capture_all_from_camera` with `capture_options` requesting only `image` → response has only `image` populated, other vectors empty (no implicit zero entries).
- `do_command` with a non-trivial `ProtoStruct` (nested map + array) → bit-identical round-trip.
- `get_object_point_clouds` with non-empty `geometries` → round-trip preserves them.
- Server-side: throwing `viam::sdk::Exception` from the mock yields a non-OK `grpc::Status` to the client (mirror `test_mlmodel`'s exception case).

### Not tested

- Auto-generated module-template files (the workflow either produces them correctly or it doesn't — asserting on generated text would just reproduce `cl_gen` logic).
- Concurrency / threading (not done for other services either).

## Risks & mitigations

- **`cl_gen` produces a broken auto-generated template.** Run `cl_gen` locally in the same Debian-bookworm + clang-15 container the workflow uses, before merging. If broken, hand-write the two `.in` files (~20 LoC each, modeled on `mlmodel.{hpp,cpp}.in`) and accept that the workflow will overwrite them on the next run (it likely won't if the input parses cleanly).
- **`Camera::raw_image` / `Camera::point_cloud` have no shared proto-conv helpers today** — Camera's client/server inline the conversion. We will lift those into `src/viam/sdk/common/private/raw_image.{hpp,cpp}` (new files) consumed by both camera and vision. The lift is mechanical and small (~30 LoC) but does mean touching `camera_client.cpp` / `camera_server.cpp` to call the new helpers; verify camera tests still pass after the lift.
- **`detection` proto's split-presence bbox fields.** Mitigated by `boost::optional` per field, `has_*` checks on both sides, and an explicit test case.
- **Server-side exception → grpc::Status mapping must match other services exactly.** Copy the catch ladder from `mlmodel_server.cpp` verbatim rather than rewriting.
- **Header install path** must match what consumers expect (`#include <viam/sdk/services/vision.hpp>`). Verify by grepping the existing install list and adding alphabetically.

## Verification (definition of done)

- `cmake --build` succeeds in the standard SDK build (Debian-bookworm container or local equivalent).
- `ctest -R test_vision` passes all assertions, including per-RPC round-trip and exception cases.
- `clang-tidy` / `clang-format` clean per repo `.clang-tidy` / `.clang-format`.
- Doxygen builds without new warnings (every public method has `///` docs matching `MLModelService::infer`).
- Manual smoke: build a trivial module from the generated template (`viam module generate` → fill stubs with `throw runtime_error` → `cmake --build`) and confirm it links against the new symbols.
- `update-templates.yml` runs cleanly on the next push to main and either no-ops (if templates were hand-written correctly) or produces the expected `vision.{hpp,cpp}.in`.

## Build sequence (one PR)

The work decomposes inside a single PR in this order — earlier steps are prerequisites for later steps:

1. `common/private/raw_image.{hpp,cpp}` — lift Camera's inline raw_image proto-conv into shared helpers.
2. Migrate `camera_client.cpp` / `camera_server.cpp` to call the new helper; confirm `test_camera` still passes.
3. `vision.hpp` (public abstract base + nested types).
4. `vision.cpp` (api/traits/operator<<).
5. `private/vision.{hpp,cpp}` (proto-conv helpers; uses raw_image helper from step 1).
6. `private/vision_client.{hpp,cpp}` + static registration.
7. `private/vision_server.{hpp,cpp}` + static registration.
8. `CMakeLists.txt` update for SDK target + header install.
9. `mocks/mock_vision.{hpp,cpp}` + `test_vision.cpp`.
10. `tests/CMakeLists.txt` update (mock added to `viamsdk_test` `target_sources`; `viamcppsdk_add_boost_test(test_vision.cpp)` appended).
11. Verify `cl_gen` produces working templates locally in the workflow's container; hand-write only if necessary.
12. PR description must explicitly call out the `point_cloud_object::point_cloud` field/type name shadow — it's intentional and matches the proto field name.

Each step is locally verifiable (compiles independently of later steps).
